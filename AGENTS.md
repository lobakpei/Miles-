# AcreMiles 開發守則

## 產品底線

- 呢個產品會影響信用卡申請、簽賬分配同里數兌換；可信度高過「答案好睇」。
- 未有足夠來源支持嘅卡、優惠、航線或條款，只可標示為「未核實」，不可進入推薦結果。
- 「基本規則通過」不等於有直航、有獎勵位或一定出到票；UI 必須分開講清楚。
- 推薦引擎係 heuristic，除非有完整最優性證明，禁止使用「最佳」、「保證最抵」等絕對字眼。
- 不可將欄位級別嘅未知資料包裝成整張卡已完全核實。

## 資料變更

1. 記錄來源、核實日期、有效期及核實層級。
2. 優惠無有效期或條款不完整時，預設隱藏，唔可以靠「expiry: null」長期展示。
3. 過期優惠要自動下架或標示已完結；香港市場日期一律以 `Asia/Hong_Kong` 計。
4. 改文章或優惠時，要掃標題、搜尋索引、圖片 alt、卡庫、FAQ、示範數據及 build marker。
5. 航線只可由已核實白名單標示為「已核實直航」；樞紐推斷只可叫「候選航司」。

## 私隱與第三方

- Google Analytics、Sentry 或其他非必要第三方 SDK 必須先取得明確同意先載入。
- 訂閱 email 只可喺用戶主動提交及同意後傳送；政策要列明實際服務供應商。
- 「重設所有資料」必須清除所有 `bm_*` 本機鍵；新增鍵時同步更新測試。
- 錯誤診斷不可傳送表單內容、email、完整 URL query、headers、cookies 或可識別使用者資料。

## 每次交付前

1. 執行 `node scripts/verify.js index.html`。
2. 確認版本號、build 日期同 `sw.js` cache 名一致。
3. 確認未核實卡唔會進入 optimizer。
4. 確認所有 RTW demo 每段核實狀態都有顯示，未核實段不可宣稱「條線齊規」。
5. 確認未同意前唔會載入 GA/Sentry。
6. 確認「重設所有資料」涵蓋程式內所有 `bm_*` key。
7. 完整資產可用時，至少做 360px 手機、768px 平板、1440px 桌面視覺 QA。

## RTW 現行真相來源

- 根目錄 `route-verification.md` 同舊交接檔屬歷史紀錄，當中「零警告／條線齊規」等結論已經過時。
- 現行狀態以 `docs/RTW-AUDIT.md`、`OW_VERIFIED`／`OW_NO_DIRECT` 同 UI 三態結果為準。

## 檔案方向

- 現階段保留單檔以減低今次修正風險。
- 下一個結構版本先拆 `data/`、`engine/`、`ui/`、`content/`，每一步都要有輸出快照比較。
- SEO 文章要有真正可索引 URL；overlay page 只可視為 app 內體驗，唔等於 SEO landing page。
