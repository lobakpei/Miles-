# 1. AcreMiles 核心思想

## 1.1 北極星信念

> **每筆消費，都值得有回報。**

呢句唔只係 slogan，而係 AcreMiles 所有產品決策嘅最高準則。

AcreMiles 唔應該被限制成：

- 一個信用卡網站
- 一個 Asia Miles 教學網站
- 一個環球票規劃器
- 一個純計算器

AcreMiles 真正係：

> **幫香港人將本來已經會發生嘅消費，轉化成更有價值嘅回報。**

現階段第一個入口係飛行里數同機票，因為旅行結果最容易令香港人產生「原來可以咁樣」嘅衝擊。將來可以發展去酒店、住宿、升級、獎品、禮券或其他 rewards，但核心問題永遠都係：

> **我呢筆消費，可以換返啲乜？**

## 1.2 產品使命

AcreMiles 唔係教人「玩里數」。

AcreMiles 要令一個本來覺得里數太麻煩、所以一直揀 cashback 嘅香港人，第一次發現：

> **原來我本來要買嘅一樣嘢，可以換返一段旅行。**

例如用戶將來介紹畀朋友時，最理想嘅說法唔係：

- 「呢個 app 幫我計到好多里數。」
- 「呢個 app 幫我揀咗張卡。」

而係：

> **「我本來買部 iPhone，最後竟然換到兩張日本機票。」**

旅行結果係主角；里數係工具；信用卡係方法；系統或 AI 只係助手。

# 2. Founder Story：點解 AcreMiles 存在

大約十年前，Founder 結婚並有二十幾萬港元婚禮支出。因為反正都要使呢筆錢，所以開始研究可唔可以換返一啲額外價值。

當時透過網上里數內容開始鑽研信用卡迎新、里數規則及出票方法，最後將消費分配到約六張信用卡，儲到二十幾萬飛行里數，並換到一套十幾程、全商務客艙、為期一個多月嘅環遊世界蜜月旅程。

最深刻嘅唔係：

- 申請六張信用卡
- 儲到二十幾萬里

而係：

> **一筆本來只會用喺婚禮嘅支出，額外換返一段正常情況下唔會用真金白銀買到嘅人生體驗。**

但整個過程非常辛苦：

- 搜集信用卡資料
- 理解迎新條件
- 分配簽帳
- 研究航空公司及聯盟
- 理解停留、轉機、開口及里數區
- 查航班與獎勵位
- 打電話同客服溝通出票

AcreMiles 嘅存在理由，就係將呢個「出票地獄」同「里數研究地獄」簡化，令普通人唔需要先變成里數專家，先可以得到旅行回報。

呢段故事可以放喺品牌故事／關於我們／社交宣傳，但首頁第一秒未必需要完整講晒。首頁首先要展示具體價值。

# 3. 目標用戶

## 3.1 最大用戶群

AcreMiles 嘅核心 target audience 係：

- 香港人
- 有正常至不錯消費能力
- 普遍鍾意旅行，而且對國家、城市及旅遊活動有一定認識
- 每年通常會旅行一次或以上
- 平時可能用 cashback
- 唔熟悉信用卡迎新、里數、航空聯盟及出票規則
- 冇興趣花大量時間學里數
- 想用最短時間睇到「我會得到啲乜」

佢哋唔係唔識旅行。佢哋可能非常清楚自己想去：

- 米蘭
- 巴黎
- 阿根廷
- 南非
- 美國
- 英國
- 北海道滑雪
- 歐洲滑雪
- 北極光
- Shopping

但佢哋唔知道、亦唔想知道：

- Oneworld
- SkyTeam
- IATA code
- 開口
- 停留
- 轉機額
- 里數分區
- 邊間航空公司應該接邊一段

所以要隱藏嘅係「里數世界嘅複雜性」，唔係旅行目的地本身。

## 3.2 用戶成熟度分布

產品設計應假設：

- 約九成主要用戶係里數新手
- 約七成主要流量會由常見消費或中等金額消費而來，例如約 HK$5,000、HK$8,000、HK$10,000、HK$15,000、HK$20,000 左右
- 約三成內容／用戶興趣可以係環球票、多城市、商務艙及進階玩法

「大額消費」唔應再成為全站預設用語，因為：

- 有啲卡簽 HK$5,000 已經有迎新
- 有啲卡係 HK$8,000
- 有啲係 HK$10,000 或以上
- 入坑關鍵唔係金額大細，而係第一次真係得到著數

建議主要用語改成：

- 「今次消費」
- 「一筆消費」
- 「準備簽賬」
- 「你本來要買嘅嘢」

避免用「大額消費」概括所有人。

# 4. 用戶轉化及商業模式

## 4.1 流量入口

AcreMiles 本身唔會自然成為第一個流量入口。主要流量會由社交平台、短片、圖文內容及生活情境宣傳帶入網站。

宣傳唔應該由信用卡或里數術語開始，而應該由生活消費開始。

錯誤方向：

- 「某信用卡迎新有 20,000 里」
- 「Asia Miles 最新分區表」

正確方向：

- 一部新 iPhone 可以變成乜旅程
- GTA 遊戲、PS5 Pro 加電視嘅整套消費，可以換到乜
- 結婚、裝修、買車等支出，可以轉化成咩旅行體驗

社交內容嘅作用係令用戶第一下產生：

> **「原來呢筆錢仲可以換返咁多？」**

## 4.2 理想轉化流程

```text
生活情境內容
↓
用戶見到具體消費與具體旅行結果
↓
產生好奇，而唔係防備
↓
進入 AcreMiles
↓
先睇示範與 big picture
↓
再輸入自己金額／情況
↓
得到個人化可行方案
↓
理解要用邊張卡、點樣完成
↓
透過 AcreMiles 申請推薦信用卡
↓
AcreMiles 獲得 referral 收入
↓
用戶第一次真實得到回報
↓
建立信任及習慣
↓
之後每次食飯、睇戲、Shopping、海外簽賬前都會諗起 AcreMiles
```

## 4.3 商業模式原則

最終收入目標係信用卡 referral／affiliate，但產品唔可以一開始畀人感覺只係想 sell 卡。

正確次序係：

1. 先證明價值
2. 再提供方案
3. 再解釋點解推薦某張卡
4. 最後先出申請入口

日後有 referral 時必須：

- 清楚披露
- 唔可以因佣金而扭曲排序
- 解釋推薦原因
- 將「適唔適合用戶」放喺「AcreMiles 賺唔賺佣」之前

Referral 係用戶成功之後自然產生嘅商業結果，唔係第一屏嘅主角。

# 5. 成功時刻及習慣循環

## 5.1 第一個 Aha Moment

第一個 Aha Moment 唔係：

- 見到一大堆里數術語
- 填完好多資料
- 學識開口／停留

而係：

> **見到一個具體消費，可以換到一個具體旅行結果。**

例如：

- 買一部 iPhone
- 原來可以換到一套短途機票

呢個係視覺及心理上嘅第一個成功。

## 5.2 真正「入坑」時刻

真正令用戶開始信任里數嘅時刻，係佢第一次實際得到著數。

唔需要定義幾多錢先叫大額，亦唔需要等到換環球票先叫成功。無論係：

- HK$5,000 簽帳食到迎新
- HK$8,000 買 iPhone 得到第一筆里數
- 第一次換到一張短途機票

只要用戶真實感受到「原來真係得」，就係入坑。

## 5.3 長期習慣目標

End goal 係用戶經常打開 AcreMiles，而唔係一年只喺結婚、裝修或買車時用幾次。

理想習慣係：

- 食飯前：用邊張卡
- 睇戲前：用邊張卡
- Shopping 前：用邊張卡
- 海外簽賬前：用邊張卡
- 買新手機前：有冇迎新值得食
- 交一筆支出前：呢筆消費可以換到乜

第一筆具體消費係入口；之後每次消費前先諗回報，先係長期產品價值。

# 6. 不可妥協嘅產品及 UX 原則

## 6.1 先展示價值，再問資料

首頁第一秒唔可以直接問：

> 「你要使幾多錢？」

對一個未建立信任嘅陌生用戶嚟講，呢種問題會令人覺得：

- 關你咩事
- 你係咪想 sell 我
- 你係咪想收集資料
- 好似呃人網站

正確做法係：

1. 先展示一個清楚、可信、可理解嘅例子
2. 用戶見到 big picture
3. 產生好奇
4. 再邀請佢輸入自己數字

原則：

> **Earn trust before asking anything.**

## 6.2 Result First

每個畫面第一層應優先回答：

- 我要簽幾多
- 我會得到幾多
- 可以換到乜
- 呢個結果同我有冇關

之後先講：

- 點樣做到
- 用邊張卡
- 第一層／第二層／第三層
- 條款
- 例外

## 6.3 Progressive Disclosure

資訊唔可以刪少，尤其涉及真金白銀、信用卡、批核、里數及出票。

但資訊要分層：

### 第一層：一眼睇明

- 消費金額
- 可得回報
- 可換旅程
- 日期／資料狀態

### 第二層：點樣做到

- 卡組合
- 迎新門檻
- 簽帳分配
- 時間表

### 第三層：完整規則

- T&C
- 排除交易
- 年費
- TU 影響
- 里數到帳
- 獎勵位
- 稅費
- 官方連結

原則：

> **Never remove information. Remove intimidation.**

## 6.4 唔好迫用戶解讀介面

用戶唔應該需要停低理解一個符號代表乜。

例如：

- 唔好只用滑雪 emoji 表示「滑雪」
- 唔好只用心形表示刪除
- 唔好用抽象圖示代替主要文字

正確做法：

- 直接寫「滑雪」
- 直接寫「刪除」但收喺適合位置
- 所有重要選項都有清楚文字

標準 SVG icon 可以保留作輔助，但必須配文字；唔可以令 emoji 成為主要語意。

## 6.5 用戶揀目標，系統處理方法

用戶應該負責講：

- 想去邊
- 想做乜
- 有幾多日
- 幾多人
- 有幾多消費／里數

系統應該負責：

- 航空公司
- 聯盟
- 機場代碼
- 轉機
- 開口
- 停留
- 里數區
- 點樣最慳

原則：

> **Users choose goals. The system chooses the path.**

## 6.6 唔好將「唔識里數」誤解成「唔識旅行」

香港用戶可以非常熟悉旅遊地方及活動，所以 Beginner flow 應該容許佢哋直接揀：

- 國家
- 城市
- 滑雪
- Shopping
- 北極光
- 沙灘
- 美食

但唔好一開始要求：

- 航空公司
- IATA code
- 停留類型
- 開口類型

## 6.7 有摩擦唔代表唔做

唔可以因為某功能會令人有防備、需要思考或較複雜，就直接話唔做。

正確問題係：

- 點樣做先唔會令人有防備
- 點樣做先可以五秒內畀價值
- 點樣由系統代替用戶處理複雜部分

產品原則唔係「Reject」，而係「Solve」。

## 6.8 美感係信任嘅一部分

AcreMiles 係 consumer product，唔係內部工具。

對目標用戶嚟講：

- 靚
- premium
- 乾淨
- 有質感
- 唔似廉價 template

本身就係可信度。

所以唔採用「先醜住驗證，之後先執靚」嘅 MVP 方法。第一版可以未完美，但必須沿用現有高質感 UI。

目前預設進入 `tab-journey`，第一個主要內容係 `planner-launch`：

# 10. 計算器第一版改法

## 10.1 導覽名稱

唔需要重寫計算引擎，但介面要逐步 reveal。

只顯示：

- 清楚寫「唔填都可以繼續」

### Step 3：幾多人一齊申請

- 一個人
- 二人家庭

### Step 4：消費點樣發生

用簡單口語：

- 一次過簽
- 分幾個月簽

完整上限解釋收喺「點解要問？」

### Step 5：偏好

- Asia Miles
- Avios
- 兩邊都比較

### Step 6：進階設定

以下預設收起：

- 最多開卡數
- 已有卡
- 其他進階條件

如現有引擎需要預設值，沿用現有安全預設，並容許用戶展開修改。

現有結果已經有 summary，但第一版要更加 outcome-first：

1. **大約可得幾多里**
2. **可以換到乜**
3. **你距離下一個更高結果差幾多**
4. **推薦卡組合**
5. **行動時間表**
6. **點樣計出嚟**
7. **完整風險／條款**

「可以換到乜」唔應放到太後。

如果有多個可達結果，優先顯示：

- 最容易理解
- 最貼近香港人常見旅程
- 唔誇大

例如短途來回、雙人短途、單程商務等。

## 10.4 多 Tier 顯示

優惠／計算結果唔好只展示最高門檻。

如同一優惠有幾級，第一屏應列出：

```text

簽 HK$20,000 → 得到 Z → 可換結果 C
```

實際金額按該優惠真實條款生成。

目的係令唔同消費能力嘅用戶都可以即刻見到「邊一級同我有關」，而唔係一見到最高 HK$60,000 就覺得唔關自己事。

# 22. 修改前備份、版本控制及回復規則

## 22.1 基準狀態

開始修改前，Work 必須提交以下資料：

- Baseline branch
- Baseline commit hash
- 原本 working tree 是否 clean
- 原本存在嘅未提交／未追蹤專案檔案清單
- 現有版本號
- 備份建立時間

如 working tree 唔 clean，必須保留現狀建立 snapshot；唔可以自行判斷某啲修改「應該唔要」。

## 22.2 三重保險

必須同時存在：

1. **Backup branch**
2. **Immutable Git tag**
3. **可下載 ZIP snapshot**

三者必須指向／包含同一個完整基準狀態。

ZIP 唔可以代替 Git backup；Git backup 亦唔可以代替產品擁有人手上嘅 ZIP。

## 22.3 備份驗證

Work 必須：

- 驗證 backup branch 與 tag 指向預期 commit
- 驗證 ZIP 可正常解壓
- 驗證 `index.html`、`sw.js`、資產、docs 及必要設定存在
- 提供 ZIP 大小及 SHA-256 checksum
- 在臨時目錄開啟備份版本做 smoke test
- 確認備份版本未依賴 feature branch 內嘅新改動先可以運作

## 22.4 Branch 保護

以下行為禁止：

- 直接 commit 到 `main`
- force-push `main`
- 刪除或改寫 backup branch
- 移動或重打同名 backup tag
- 將未完成版本 merge 入 `main`
- 自動 deploy／publish
- 為咗方便重構而刪除現有功能或資料
- 對 `localStorage` schema 做不可逆修改而冇 migration／fallback

## 22.5 回復能力

完成 Draft 前，Work 必須證明可以回復到原始版本，並提供最簡單嘅 rollback 指示，包括：

- 點樣由 backup branch 開返原版
- 點樣由 immutable tag 建立新 branch
- 點樣用 ZIP 在新資料夾恢復
- 如 feature branch 失敗，點樣保留而唔污染 `main`

**未經產品擁有人明確指示，不得對 `main` 執行 destructive reset。**

## 22.6 完成交付必須包含

- Baseline commit hash
- Backup branch 名稱
- Backup tag 名稱
- ZIP 檔名
- ZIP SHA-256 checksum
- Feature branch 名稱
- Draft PR
- 修改檔案清單
- 主要截圖
- 測試結果
- 已知限制
- 待核實數據
- Rollback 步驟

# 25. 最終判斷準則

每一個新畫面唔係問「做唔做」，而係問：

1. 點樣先可以先畀用戶見到價值，而唔令佢有防備？
2. 點樣先可以由結果開始，而唔係由規則開始？
3. 點樣先可以保留完整資訊，但唔喺第一眼嚇走人？
4. 點樣先可以由系統處理複雜部分，而唔係逼用戶先學識？
5. 呢個畫面有冇令用戶更接近以下一句？

> **「我本來只係買一樣嘢，最後竟然換到一段旅行。」**

如果答案未夠清楚，唔係刪功能，而係再設計得更直接、更簡單、更可信。
